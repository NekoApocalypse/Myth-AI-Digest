## 🔬 1. Microgpt — A Complete GPT in 200 Lines of Pure Python

**Source:** Andrej Karpathy · karpathy.github.io · Feb 12, 2026
**HN:** https://news.ycombinator.com/item?id=47202708 · ⬆ 1,677 pts

Karpathy's latest educational artifact: `microgpt`, a single self-contained Python file (~200 lines, zero dependencies) that trains and runs a GPT from scratch. It is the terminus of a decade-long series — micrograd → makemore → nanoGPT — distilled to its irreducible core.

**What's inside:**

- **Custom autograd engine** (`Value` class): scalar-level backprop with topological sort, implementing `+`, `*`, `**`, `log`, `exp`, `relu`. Algorithmically identical to PyTorch's `.backward()`, just on scalars.
- **Tokenizer:** character-level (27 tokens: a–z + BOS), simple but complete.
- **GPT-2-like architecture:** token + position embeddings, multi-head self-attention with explicit KV cache (built step-by-step, not hidden inside vectorized ops), RMSNorm, ReLU MLP, residual connections, linear output head.
- **Adam optimizer:** first and second moment buffers, bias correction, linear learning rate decay.
- **Training loop + inference:** trains on 32K baby names; loss drops from ~3.3 (random) → ~2.37 in 1,000 steps.

**Why it matters:** The KV cache is made explicit even during training — an unusual choice that reveals it was conceptually always there. Karpathy also provides a staged `build_microgpt.py` gist showing the progression from a bigram count table (train0) up to the full Adam-trained GPT (train5), diffs included. The "real stuff" section clearly maps each microgpt component to its production counterpart (BPE, tensors, FlashAttention, SFT, RL post-training). Runnable in ~1 minute on a MacBook with `python train.py`.

## 🌐 2. WebMCP — Chrome Proposes a Browser-Native Agent API Standard

**Source:** Chrome for Developers · developer.chrome.com · Feb 10, 2026
**HN:** https://news.ycombinator.com/item?id=47211249 · ⬆ 93 pts

Google Chrome has opened an early preview program for **WebMCP**, a proposed web standard that makes websites "agent-ready" via two new browser APIs:

- **Declarative API:** Standard actions defined directly in HTML forms — agents fill them without needing to interpret DOM layout.
- **Imperative API:** Complex, dynamic interactions requiring JavaScript execution, for workflows that can't be expressed declaratively.

The stated goal is to replace brittle DOM actuation (raw clicking/scraping) with structured, intent-expressing interfaces. Use cases: booking systems, ecommerce checkout, customer support ticketing. The pitch: when a site defines its own tools, agents can act on it "with increased speed, reliability, and precision."

**Significance:** This is Google positioning WebMCP as an alternative/complement to Anthropic's MCP protocol — but at the browser layer rather than the LLM-tool layer. If adopted, it would allow any website to expose agent-callable affordances without requiring a separate MCP server or API. Whether this gains traction over raw browser automation (Playwright, computer-use) or server-side MCP is an open question.

## ⚔️ 3. MCP is Dead. Long Live the CLI.

**Source:** Eric Holmes · ejholmes.github.io · Feb 28, 2026
**HN:** https://news.ycombinator.com/item?id=47208398 · ⬆ 234 pts · 157 comments

A pointed technical argument that MCP provides no real-world benefit for agentic coding workflows, and that well-designed CLIs are strictly superior for most use cases.

**Core arguments:**

1. **LLMs already know CLIs.** Trained on millions of man pages and shell scripts, frontier models use `gh`, `kubectl`, `aws` etc. natively — no new protocol needed.
2. **Composability is the killer argument.** CLIs pipe through `jq`, `grep`, file redirects. MCP cannot. Example: `terraform show -json plan.out | jq '...'` — doing this with MCP means either dumping an entire plan into context (expensive, often impossible) or pre-building custom filtering into the server.
3. **Debuggability.** CLI tools are reproducible outside the agent. MCP failures require spelunking through JSON transport logs.
4. **Auth is a solved problem.** `gh auth login`, `aws sso login`, `kubeconfig` — battle-tested, no MCP-specific troubleshooting. Re-auth in multi-MCP setups is a recurring pain.
5. **No background process overhead.** CLIs are binaries on disk. MCP servers are processes that can hang, fail to start, or require state management.

**Where MCP still makes sense (per author):** When no CLI equivalent exists; for non-developer users of chat-style interfaces who can't run a terminal; for services requiring OAuth-style auth delegation; as an internal tool gateway at enterprise scale.

**HN consensus:** Split but substantive. Many practitioners agree CLIs are better for power users and coding agents (Claude Code, OpenClaw), while defending MCP for non-technical users, remote/cloud deployments, and fine-grained permission scoping. The strongest counterargument: MCP's Streamable HTTP with OAuth discovery is the right model for consumer-facing agents that run in web UIs — CLIs simply aren't accessible there.

## 🏷️ 4. Why XML Tags Are So Fundamental to Claude

**Source:** glthr.com · Feb ~28, 2026
**HN:** https://news.ycombinator.com/item?id=47207236 · ⬆ 153 pts · 107 comments

An article arguing that XML is not an arbitrary choice for Claude prompting — it's structural to how the model was trained and how it processes context.

**Key points from article + HN discussion:**

- **Tool-calling origin:** Anthropic's original tool-calling API exposed tool invocations as XML (`<invoke name="Read"><parameter name="file_path">…</parameter></invoke>`). The model was explicitly fine-tuned on this format before the JSON API was introduced; internally, tool calls are still likely templated as XML before reaching the model context.
- **Namespace isolation:** The `antml:` namespace prefix ensures user-provided XML tags can't collide with internal tool-call XML, providing a structural defense against certain prompt injection vectors.
- **Balanced-bracket disambiguation:** Named closing tags (`</instructions>`) help the model track nesting depth across long contexts — a genuine advantage over JSON (requires counting brackets) or Markdown (section ends are implicit).
- **Training corpus alignment:** HTML is one of the largest components of pre-training data. The model has processed XML/HTML structure far more than Markdown and has stronger in-context disambiguation of nested structures.
- **`<think>` precedent:** The practice of using `<think>` tags to elicit extended reasoning predates official reasoning models (it worked as far back as Sonnet 3.0) and remains a usable pattern today.

**Caveats raised by HN:** Several practitioners report no observable difference using XML vs. clean Markdown in modern Sonnet/Opus models, suggesting the advice may be stale or context-dependent. The article covers prompting harnesses/automation, not end-user chat.
